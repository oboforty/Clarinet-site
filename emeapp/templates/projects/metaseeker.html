MetaSeeker

MetaSeeker is a derivative metabolome database developed in coordination with Uppsala University's Komorowski Lab. It's built upon the major chemical databases used for metabolomics by automatically mapping their records and allowing manual curation (re-mapping & editing entries) for authorized users.

MetaSeeker is a mixture of Single-page apps and old-school backend-rendered pages. Complex frontend interfaces rely on Vue.js SAP, while simple pages such as legal and manuals are rendered using Flask's built in Jinja2. Users are able to search metabolites through several web interfaces;

String Search

Regular search is achieved using a custom search module built in Eme, which allows text search for several metabopte attributes (e.g. chemical names, ascii structure, chemical formula, description).

Molecular Search

This is a method of search for several cheminformatics websites. Using JSME - a Javascript port of JME, a popular chemichal editor - the user constructs the molecule that is in the question, which is then converted to InChI format using a Javascript port of the InChI C algorithm. This ascii string is then searched in the database. An alternative method for InChI-based search is creating an MDL file and issuing a custom similarity-search in the database. This latter method is not implemented yet, but is a planned feature.

User Authentication

Just like the bulk of my projects, MetaSeeker uses the Doors Authentication system over OAuth2 bearer token and web authorization code grant type.

Discovery Search

Other than relying on the regular cached search, users can directly trigger the custom algorithm that used in constructing the derivative database. The `discovery` algorithm is responsible for mapping identifiers between the external database. When the user searches a term (regular search) that is not found in the cache, the discovery algorithm is triggered automatically ands its result, the universal metabolite record is then cached in the search engine.

Building the database

Several Eme tasks and commands are responsible for constructing the database in consecutive steps. The relevant attributes of external database records are stored individually and their foreign key mapping is stored in an additional table. A future planned feature is to assign a custom, universal metabolome identifier, so that it could be used as a single source of truth in metabolomics research.
